MODEL_MANIFEST_AUDIT

1. RESUMEN

Objetivo de la auditoría:
- Analizar si ApiModels tiene ya información suficiente para publicar en el futuro un manifest técnico de modelos, por ejemplo GET /models/manifest.
- Comparar esa información con lo que el Backend guarda hoy en MongoDB mediante IssueModel y con el mapeo actual hacia endpoints de ApiModels.
- Proponer una estructura JSON y una estrategia segura de sincronización futura.

Resultado principal:
- ApiModels ya tiene una base muy aprovechable para un manifest: registry central, endpoints, schemas Pydantic, ejemplos de request/response, operationId, summary, description y contrato común.
- Esa información no es suficiente todavía para sincronizar automáticamente IssueModel de Mongo de forma completa y segura.
- Faltan metadatos funcionales y de aplicación: displayName estable, aliases, role del modelo, si debe ser visible como IssueModel, evaluationStructure, supportedDomains, parámetros en formato compatible con Mongo, restricciones de UI, compatibilidad con escenarios y ciclo de vida.
- Backend integra como modelos de issue: TOPSIS, Fuzzy TOPSIS, BORDA, ARAS y Herrera-Viedma CRP.
- ApiModels publica además /bwm y /cmcc.
- BWM está integrado en Backend como servicio auxiliar para pesos, no como IssueModel.
- CMCC está publicado en ApiModels, pero no aparece integrado en Backend/modelCatalog ni en seedIssueModels.js.

Conclusión:
- La fase 1 más segura no debería sincronizar Mongo automáticamente.
- Primero conviene publicar un manifest read-only, derivado del registry y enriquecido con metadatos manuales mínimos.
- Después, Backend debería consumirlo en modo dry-run/comparación.
- La escritura en Mongo debería llegar más adelante, actualizando solo campos técnicos y sin borrar ni pisar contenido editorial/admin.

  - Usa data.weights para actualizar issue.modelParameters.weights.

Campos que ApiModels puede publicar ya mismo sin inventar nada:

Nivel manifest:
- apiName: se puede tomar de FastAPI title: "CreteValley Decision Models API".
- apiVersion: se puede tomar de FastAPI version: "1.0.0".
- contract: success/message/data/error.
- validationErrorCode: VALIDATION_ERROR, ya usado en core/application.py.
- endpoints base publicados: deducibles de MODEL_REGISTRY.

Nivel modelo, desde ModelRouteRegistration:
- key: registration.name.
- path: registration.path.
- method: "POST", deducido del router actual.
- requestModel: registration.request_model.__name__.
- summary: registration.summary.
- description: registration.description.
- operationId: registration.operation_id.
- responseExamples: registration.response_examples.

Nivel schema, desde Pydantic:
- requestSchema completo mediante model_json_schema().
- campos requeridos.
- propiedades.
- tipos JSON básicos cuando Pydantic los expone.
- descriptions de campos.
- defaults.
- request example desde json_schema_extra.

Nivel validación:
- Validaciones semánticas existen en código Pydantic.
- Algunas reglas pueden inferirse textualmente por ValueError, pero no están declaradas como metadata estructurada.
- Reglas claramente detectadas:
  - matrices no vacías.
  - misma forma entre matrices de expertos.
  - criterionTypes con longitud igual al número de criterios.
  - weights con longitud igual al número de criterios.
  - fuzzy values deben ser ternas [l, m, u].
  - BWM mic/lic deben existir, tener misma longitud y ser consistentes entre expertos.
  - CMCC o, c, omega y w deben tener la misma longitud.
  - Herrera-Viedma usa matrices cuadradas y misma forma.

Nivel salida:
- Se pueden deducir dataKeys desde response_examples:

  - ARAS: collective_matrix, matrix_used, collective_scores, collective_ranking, plots_graphic.

5. CAMPOS FALTANTES O PENDIENTES

Campos necesarios para un manifest completo que ApiModels no tiene hoy de forma estructurada:

- displayName:
  Nombre humano que debe coincidir o mapearse con Mongo/Frontend.
  Ejemplo: registry usa herrera_viedma_crp, seed usa "Herrera Viedma CRP", modelCatalog acepta también "HERRERA-VIEDMA CRP".

- aliases:
  Lista de nombres aceptados por Backend.
  Actualmente solo existen en Backend/modelCatalog.js.

- role:
  Clasificación técnica/funcional:
  - issueModel.
  - weightingService.
  - consensusModel.
  - utilityModel.
  - legacy/experimental.
  Esto es necesario para no convertir BWM o CMCC en modelos seleccionables por accidente.

- publicInIssueCatalog:
  Booleano o estado equivalente para indicar si debe sincronizarse como IssueModel.

- evaluationStructure:
  direct o pairwiseAlternatives.
  Hoy está en Mongo/seed y Backend, no en ApiModels registry.

- isConsensus:
  Hoy está en Mongo/seed.
  Puede inferirse parcialmente por nombre/salida cm, pero no debe inventarse.

- isMultiCriteria:
  Hoy está en Mongo/seed.
  Puede inferirse parcialmente por matrices y criteria, pero no está declarado.

- supportedDomains:
  numeric/linguistic y restricciones de UI.
  Hoy está en Mongo/seed.

- parameters en formato IssueModel:
  ApiModels schemas saben qué payload acepta la API, pero no publican la estructura exacta de IssueModel.parameters:
  - name.
  - type number/array/fuzzyArray.
  - default.
  - restrictions min/max/step/length/sum/allowed.

- modelParameters metadata:
  Necesario separar parámetros del endpoint de parámetros configurables por el admin/usuario.
  Ejemplo:
  - TOPSIS/ARAS/Fuzzy TOPSIS usan modelParameters.weights.
  - Herrera usa ag_lq, ex_lq, b, beta.
  - BWM usa eps_penalty pero Backend lo hardcodea en 1 y no es IssueModel.
  - CMCC tiene o, c, omega, w, eps, mu0, lower_bound, upper_bound.

- inputKind:
  Tipo de input esperado:
  - directCrispMatrix.
  - directFuzzyMatrix.
  - pairwisePreferenceMatrix.
  - bwmExpertComparisons.
  - cmccOpinionVector.

- outputKind:
  Tipo de salida:
  - ranking.
  - consensusRanking.
  - weights.
  - adjustedConsensusOpinions.

- supportsScenarios:
  Backend escenarios solo soporta ciertos modelos según evaluationStructure y output adapters.

- scenarioOutputAdapter:
  Backend construye outputs de forma distinta si modelKey === herrera_viedma_crp.
  Ese comportamiento podría declararse para evitar hardcode.

- criterionTypes:
  ApiModels acepta max/min y alias benefit/cost.
  Falta metadata estructurada que indique canonicalValues y aliases.

- modelVersion:
  No existe versión por algoritmo.

- availability:
  No existe available/deprecated/stale/experimental.

- deprecation/legacy:
  CMCC y legacy no están claramente clasificados para manifest.

- moreInfoUrl:
  Existe en seed de Backend, no en ApiModels.

- textos editoriales largos:
  smallDescription y extendDescription existen en seed/Mongo, no en ApiModels registry.

- ownership:
  No está declarado qué campos puede sincronizar Backend y cuáles debe preservar de Admin.

8. CAMPOS QUE DEBERÍAN VENIR DE APIMODELS

Campos técnicos que deberían venir de ApiModels:

- key:
  Identificador estable del modelo en ApiModels.
  Ejemplo: topsis, aras, borda.

- endpoint.path y endpoint.method:
  ApiModels es la fuente real del endpoint ejecutable.

- operationId:
  Ya existe en registry.

- request schema:
  ApiModels conoce el payload real aceptado.

- request example:
  Ya existe en schemas/model_requests.py.

- response examples:
  Ya existen en registry/model_registry.py.

- response dataKeys:
  Deben derivarse o declararse desde ApiModels porque dependen del output real.

- accepted criterionTypes:
  ApiModels acepta max/min y aliases benefit/cost.

- modelParameters técnicos:
  ApiModels debería declarar qué parámetros espera ejecutar.
  Ejemplos:
  - weights crisp para TOPSIS y ARAS.
  - weights fuzzy para Fuzzy TOPSIS.
  - ag_lq, ex_lq, b, beta para Herrera-Viedma.
  - eps_penalty para BWM.
  - o, c, omega, w, eps, mu0, lower_bound, upper_bound para CMCC.

- validation constraints técnicas:
  Ejemplos:
  - length matchCriteria.
  - fuzzy triplets.
  - matrices square para pairwise.
  - same shape across experts.

- inputKind:
  ApiModels sabe si consume matrices crisp, fuzzy, pairwise, datos BWM o vectores CMCC.

- outputKind:
  ApiModels sabe si produce ranking, pesos, consenso, ajustes, etc.

- availability/status técnico:
  ApiModels sabe qué endpoints están publicados.

- apiVersion/modelVersion:
  Debería venir de ApiModels cuando exista.

- role técnico:
  Debería venir de ApiModels, al menos para evitar sincronizar como IssueModel endpoints auxiliares como BWM.

  No existe en schema actual. Si se añade, debe poder controlarlo Admin/Backend.

- smallDescription:
  Puede inicializarse desde manifest, pero no debería pisarse automáticamente si Admin lo edita.

- extendDescription:
  Igual que smallDescription.

- moreInfoUrl:
  Actualmente está en seed con URLs de ejemplo. Puede ser editorial/aplicación.

- traducciones o textos orientados a usuario:
  ApiModels puede aportar documentación técnica, pero la UI puede necesitar textos curados.

- reglas de visibilidad por rol, tenant o entorno:
  Propias de Backend/Admin.

- supportedDomains como política UX:
  La capacidad técnica puede venir de ApiModels, pero el rango exacto de dominios mostrado en UI puede ser una decisión de la app.
  Ejemplo: TOPSIS en seed limita numeric range 0..1, aunque el algoritmo podría aceptar otros valores normalizados. Esto es pendiente de confirmar.

- compatibilidad con issues existentes:
  Backend debe decidir si una actualización técnica se aplica a modelos con issues activos o solo a nuevos issues.

- aliases legacy de negocio:
  ApiModels debería publicarlos si quiere ser fuente técnica, pero Backend debe preservar compatibilidad con datos existentes.

- ownership de campos:
  Backend debe guardar qué campos son administrados por manifest y cuáles por Admin.


10. RIESGOS DE SINCRONIZACIÓN

Riesgos principales:

- Pisar descripciones editadas manualmente:
  Si el sync hace $set completo como seedIssueModels.js, puede reemplazar textos curados por Admin.

- Crear modelos auxiliares como modelos de issue:
  BWM y CMCC están publicados en ApiModels, pero no son IssueModel hoy.
  Sin role/publicInIssueCatalog, podrían aparecer indebidamente en la UI.

- Borrar modelos desaparecidos:
  Si ApiModels deja de publicar un endpoint temporalmente, borrar el IssueModel rompería issues históricos.

- Cambios de nombre:
  "Herrera Viedma CRP", "HERRERA-VIEDMA CRP" y "CRP" apuntan al mismo endpoint.
  Sin key estable y aliases, se pueden crear duplicados.

- Cambios de parámetros:
  Si weights o restricciones cambian automáticamente, issues activos o escenarios podrían quedar inválidos.

- Cambios de supportedDomains:
  Puede afectar al wizard de creación, dominios de expresión y escenarios.

- Diferencia entre schema API y parámetros configurables:
  El request schema completo no equivale a IssueModel.parameters.
  Ejemplo: matrices y criterionTypes son input runtime, no parámetros que el usuario configure como modelParameters.

- Resultados con estructuras distintas:
  Backend tiene adapters específicos para direct vs herrera_viedma_crp.
  Si un nuevo modelo devuelve otro shape, escenarios/resolución pueden fallar.

- CMCC sin integración Backend:
  ApiModels lo publica, pero Backend no sabe resolverlo como IssueModel.

- BWM como servicio auxiliar:
  Forma parte del flujo de pesos, no del catálogo de modelos seleccionables.

- Inconsistencias entre seed y DB real:
  El seed muestra lo esperado, pero la colección real puede haber sido modificada.
  Estado real de Mongo: pendiente de confirmar.

- Sin versionado:
  No hay modelVersion por algoritmo ni manifestVersion.

- Sync en caliente:
  Cambiar catálogo mientras usuarios crean issues puede generar experiencias inconsistentes.

- Validaciones no estructuradas:
  Parte de las reglas están en validators Python como ValueError, no como metadata machine-readable.


11. ESTRATEGIA SEGURA DE SINCRONIZACIÓN

Estrategia recomendada:

1. Publicar manifest read-only.
- Sin escribir en Mongo.
- Debe incluir key, endpoint, request schema, examples y metadata manual mínima.

2. Añadir dry-run en Backend.
- Comparar manifest contra IssueModel.
- Reportar:
  - modelos nuevos.
  - modelos sin endpoint.
  - diferencias técnicas.
  - modelos Mongo no presentes en manifest.
  - endpoints ApiModels no sincronizables como IssueModel.

3. Usar key estable.
- No sincronizar por name únicamente.
- Recomendado futuro: añadir a Mongo un campo como manifestKey o apiModelKey.
- Mientras no exista, mapear con aliases de forma explícita.

4. Crear solo modelos marcados como publicInIssueCatalog=true.
- No crear BWM.
- No crear CMCC salvo que el manifest lo declare explícitamente y Backend tenga adapter.

5. Actualizar solo campos técnicos.
- Candidatos:
  - api endpoint key/path.
  - evaluationStructure.
  - isConsensus.
  - isMultiCriteria.
  - parameters técnicos.
  - supportedDomains técnicos.
- Pero incluso estos deben revisarse con cuidado si hay issues activos.

6. Preservar campos Admin/editoriales.
- No pisar:
  - smallDescription.
  - extendDescription.
  - moreInfoUrl.
  - orden/visibilidad si se añaden.

7. No borrar modelos automáticamente.
- Si un modelo desaparece del manifest:
  - marcar como stale/unavailable.
  - impedir nuevos issues si corresponde.
  - mantener histórico.

8. Mantener auditoría.
- Guardar fecha de última sincronización.
- Guardar manifestVersion/modelVersion usado.
- Guardar diferencias aplicadas.

9. Aplicar modo allowlist inicialmente.
- Solo permitir sync de TOPSIS, ARAS, BORDA, Fuzzy TOPSIS y Herrera-Viedma CRP.
- Dejar BWM y CMCC fuera de IssueModel salvo decisión posterior.

10. Validar antes de escribir.
- Validar que parameters del manifest caben en parameterSchema.
- Validar que evaluationStructure pertenece al enum.
- Validar que supportedDomains tiene forma compatible.
- Validar que getModelEndpointKey o su reemplazo sabe resolver el key.


12. FASE 1 RECOMENDADA

Fase 1 más segura:

1. Añadir metadata manual mínima al registry de ApiModels.
- Sin cambiar ejecución de modelos.
- Campos mínimos sugeridos:
  - displayName.
  - aliases.
  - role.
  - publicInIssueCatalog.
  - evaluationStructure.
  - isConsensus.
  - isMultiCriteria.
  - inputKind.
  - outputKind.
  - supportsScenarios.
  - parameters en formato compatible con IssueModel.
  - supportedDomains.

2. Crear endpoint read-only GET /models/manifest.
- Preferiblemente en router separado.
- Debe leer MODEL_REGISTRY y devolver data.models.
- Debe usar el contrato success/message/data/error.
- No debe consultar Mongo.
- No debe depender del Backend.

3. Añadir tests de manifest en ApiModels.
- Verificar que todos los modelos registrados aparecen.
- Verificar que BWM y CMCC no se marcan como IssueModel si no corresponde.
- Verificar que TOPSIS/ARAS/BORDA/Fuzzy TOPSIS/Herrera tienen metadata mínima.
- Verificar ausencia de msg/results.

4. Añadir script/servicio Backend en modo dry-run.
- Consumir /models/manifest.
- Comparar con IssueModel.
- No escribir.
- Generar reporte.

5. Solo después plantear sync real.
- Upsert controlado.
- No borrar.
- No pisar campos editoriales.
- Marcar stale/unavailable.

Decisión pendiente antes de implementar sync real:
- Confirmar si CMCC debe ser público, legacy o experimental.
- Confirmar si BWM debe aparecer solo como weightingService.
- Confirmar si supportedDomains debe ser fuente técnica de ApiModels o política de UI en Backend.
- Confirmar cómo tratar modelos existentes con issues activos.


13. FICHEROS IMPORTANTES PARA IMPLEMENTARLO

ApiModels:

- ApiModels/registry/model_registry.py
  Fuente principal para construir el manifest.
  Ya contiene key, path, request_model, handler, summary, description, operationId y ejemplos.
  Sería el lugar natural para añadir metadata manual por modelo.

- ApiModels/schemas/model_requests.py
  Contiene schemas Pydantic, ejemplos de request y validaciones semánticas.
  Permite generar request.schema y documentar inputs.

- ApiModels/schemas/common.py
  Define ApiError y ModelExecutionResponse.
  Para manifest convendría crear en el futuro un schema específico o generalizar ModelExecutionResponse a una respuesta común reutilizable.

- ApiModels/services/model_handlers.py
  Permite conocer handlers y data real devuelta por cada modelo.
  No debería ser fuente principal del manifest, pero ayuda a documentar outputKind/dataKeys.

- ApiModels/api/routers/models.py
  Router actual de endpoints POST dinámicos.
  Puede alojar un GET /models/manifest en fase mínima, aunque se recomienda router separado.

- ApiModels/core/application.py
  Incluye routers en la app.
  Si se crea router separado de manifest, debe incluirse aquí.

- ApiModels/models/*/*_model.py
  Implementaciones reales.
  Útiles para confirmar output real y limitaciones técnicas.

Backend:

- Backend/models/IssueModels.js
  Define el shape Mongo que el sync debería respetar.

- Backend/seeds/seedIssueModels.js
  Muestra el catálogo base esperado hoy por la app.
  Importante para migrar hacia manifest sin perder metadata existente.

- Backend/services/modelApi/modelCatalog.js
  Mapeo actual nombre -> endpoint.
  En el futuro podría reemplazarse o generarse desde manifest, pero hoy es la fuente de integración.

- Backend/services/modelApi/modelParamNormalizer.js
  Normalización genérica de parámetros enviados a ApiModels.
  El manifest puede ayudar a declarar parámetros especiales.

- Backend/services/modelApi/modelResponse.js
  Adaptador actual del contrato success/message/data/error de ApiModels.

- Backend/modules/issues/issue.resolution.js
  Muestra qué necesita Backend para resolver issues: endpoint key, matrices, criterionTypes, modelParameters y output shape.

- Backend/modules/issues/issue.scenarios.js
  Muestra qué metadata necesita Backend para escenarios: evaluationStructure, supportedDomains, parameters, endpoint key y adapters de salida.

- Backend/modules/issues/issue.weights.js
  Muestra integración especial con BWM como servicio auxiliar y no como IssueModel.

- Backend/modules/issues/issue.evaluationStructure.js
  Define direct y pairwiseAlternatives y mantiene compatibilidad con isPairwise legacy.

- Backend/modules/issues/issue.creation.js
  Usa IssueModel por name durante creación de issues.
  Cualquier sync debe preservar compatibilidad con nombres existentes.

Frontend:

- No se recomienda tocar Frontend en fase 1.
- A futuro, el Frontend consume modelos vía Backend /issues/models y espera campos como smallDescription, supportedDomains, parameters y evaluationStructure.
